## 🎯 Architecture Highlights

### Clean Architecture Implementation

**Separation of Concerns**:
- ✅ Domain layer has zero Flutter dependencies (pure Dart)
- ✅ Data layer implements domain interfaces
- ✅ Presentation layer depends on domain abstractions
- ✅ Clear dependency flow: Presentation → Domain ← Data

**SOLID Principles**:
- ✅ **S**ingle Responsibility: One use case per action
- ✅ **O**pen/Closed: RecordingStrategy open for extension
- ✅ **L**iskov Substitution: All repo impls substitutable
- ✅ **I**nterface Segregation: Thin, focused interfaces
- ✅ **D**ependency Inversion: Domain owns abstractions

**Key Patterns**:
- ✅ Repository Pattern: Domain interfaces, data implementations
- ✅ Strategy Pattern: LiveChunk vs RecordFull modes
- ✅ Result Monad: Functional error handling
- ✅ GetX Mixins: DisposableMixin, LoggerMixin, ReactiveStateMixin

### Database Design (Drift)

**4 Tables with Relationships**:
```sql
recordings (id, title, mode, status, client_code, metadata, ...)
  ├─ segments (id, recording_id, index, started_at, paused_at, ...)
  │    └─ chunks (id, recording_id, segment_id, status, hash, ...)
  └─ upload_jobs (id, recording_id, status, progress, ...)
```

**Chunk State Machine**:
```
pending → encrypting → ready → uploading → uploaded → confirmed
                                  ↓
                                failed → (retry in v2)
```

**Reactive Streams**:
- ✅ `watchAllRecordings()` for live UI updates
- ✅ `watchRecording(id)` for single recording updates
- ✅ `watchChunksForRecording(id)` for chunk progress

### Recording Flow

**Two Modes Implemented**:

1. **Live-Chunk Mode**: 
   - Record → chunk → encrypt (isolate) → hash → enqueue → upload
   - Paused sessions continue processing already-captured chunks

2. **Record-Full Mode**:
   - Record full session → stitch → encode MP3 → encrypt → hash → upload
   - Paused sessions sit idle

**Simultaneous Sessions**:
- One active recording maximum
- Multiple paused recordings allowed
- Manual pause required before starting/resuming another

## 🎓 Key Decisions & Rationale

| Decision | Rationale |
|----------|-----------|
| **GetX over BLoC** | Faster scaffolding for solo dev; Clean layers enable future BLoC migration |
| **Drift over Isar** | Relational semantics for queue, FK constraints, transactions, migrations |
| **Dart crypto over native** | Reuse production app's proven Dart implementation; avoid cross-platform sync issues |
| **Plugin recording** | Session lengths (<2 hrs) don't require native; graduate only if soak fails |
| **No TDD in v1** | Timeline constraint; pure Dart domain enables v2 test coverage addition |
| **Result<T> monad** | Functional error handling without try-catch clutter |
| **V1/V2 boundaries** | Clear interface stubs prevent scope creep; real impls in v2 |

## 🔧 Setup Instructions

### 1. Install Dependencies
```bash
cd scribble_project
flutter pub get
```

### 2. Generate Drift Code (REQUIRED)
```bash
flutter pub run build_runner build --delete-conflicting-outputs
```

This generates `lib/app/data/database/database.g.dart` with the actual drift implementation.

### 3. Run on Device
```bash
# iOS
flutter run -d ios

# Android
flutter run -d android
```

### 4. Verify Architecture
- App should launch to login screen
- Tap "Continue" → Home screen with two mode buttons
- Tap a mode → Recording screen (will attempt to start recording)
- Database queries should work after drift code generation
